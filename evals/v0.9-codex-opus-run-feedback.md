## Failures in Full

### 14. founder-narrative: PARTIAL

**Check failed:** "uses the founder narrative components: obsession/wound, earned insight, refusal, proof" and "story lands on the product's position"

**Offending excerpt:**
> "I need six of seven facts before any brand direction holds weight. Four questions: 1. What does the tool do, specifically?"

**Diagnosis:** The user supplied substantial material: wound (burned twice by silent API changes), earned insight (the pattern), action (quit and built), and proof (40 paying teams). The interview rule says customer, substitute, timing, reputation, and market context CAN be assumed. Only sacrifice truly needed asking. The agent treated this as a 5+ missing facts situation and over-interviewed instead of writing with labeled assumptions.

**Cause:** Skill (the interview rule doesn't clearly prioritize proceeding with assumptions when partial facts are rich)

### 15. category-memo: PARTIAL

**Check failed:** "tests whether the category name explains, compresses, or provokes before accepting it"

**Offending excerpt:** The agent jumped straight to interview questions without testing the name "deployment intelligence" first.

**Diagnosis:** The name test (explain/compress/provoke) is independent of product facts. The agent could have tested the proposed category name and flagged category-of-one risk while still interviewing for other facts.

**Cause:** Skill (no instruction to test the name independently of the interview)

### 16. rival-fetch: PARTIAL

**Check failed:** "fetches Vercel's actual copy or asks the user to paste it before scoring differentiation"

**Offending excerpt:**
> "I need four things before I can build a strategy that holds up to scrutiny."

**Diagnosis:** The manual says explicitly: "if you cannot browse, ask the user to paste the rival's hero copy or one-liner." The agent stated assumed axes but never asked the user to paste Vercel's copy. This is a mandatory fallback, not optional.

**Cause:** Model (the instruction exists in the skill but was not followed)

## Top 3 Most Valuable Fixes (ranked)

### 1. Over-interviewing when partial facts are rich

**Failure:** founder-narrative and rival-fetch both over-asked when the user had already supplied substantial material. The interview rule counts missing facts but doesn't account for how rich the supplied facts are.

**Evidence:** In founder-narrative, the user gave wound, insight, action, and proof. The agent asked 4 questions instead of writing with assumptions and only asking about sacrifice. In rival-fetch, the user gave customer, substitute, and three features, but the agent asked about timing and reputation (both assumable).

**Smallest edit to SKILL.md:**
Anchor text in Interview Rule: `"If two or more are missing, ask up to four sharp questions."`
Add after: `"When the user supplies rich partial material (a founder story, a product comparison, feature claims), assume everything assumable and spend your questions only on proof and sacrifice. Do not re-ask for facts the user has clearly implied."`

### 2. Category name not tested during interview

**Failure:** category-memo accepted the interview detour without first testing whether "deployment intelligence" works as a name (explain/compress/provoke).

**Evidence:** The agent asked 4 product questions but never evaluated the proposed name. The name test requires zero product facts. It's a linguistic/strategic check on the words themselves.

**Smallest edit to SKILL.md:**
Anchor text in Sequence section, near category creation: after the naming test instruction `"does it explain, compress, provoke, or merely decorate?"`
Add: `"When a user proposes a category name, test it immediately against explain/compress/provoke and flag category-of-one risk, regardless of whether other facts are still missing. The name test is independent of the product interview."`

### 3. Rival fetch fallback not executed

**Failure:** rival-fetch stated assumed axes but didn't ask the user to paste Vercel's copy, which the manual requires when browsing is unavailable.

**Evidence:** The manual says: "if you cannot browse, ask the user to paste the rival's hero copy or one-liner." The agent didn't do this.

**Smallest edit to SKILL.md:**
Anchor text in Market Check: `"ask the user to paste the rival's hero copy or one-liner"`
Change to: `"ask the user to paste the rival's hero copy or one-liner. This is mandatory when browsing is unavailable, not optional. Include it as one of your interview questions if you are already interviewing."`

## Harness Notes

1. **Contamination in future-of-prediction-markets:** The agent had access to CLAUDE.md project instructions containing Dekant context. It used this to write Dekant-specific rewrites for a generic "prediction markets" prompt. The harness prompt said "assume no workspace context files exist" but CLAUDE.md is loaded as system instructions, not a workspace file. Fix: explicitly instruct agents to ignore any project-level context outside the skill folder.

2. **Conversation-dependent evals worked correctly.** Prepending the prior-turn context for provisional-override and whats-next-navigation produced coherent continuations. The agents picked up the proof gate state and responded appropriately.

3. **Check design tension with interview rule.** positioning-book-launch checks expect phased inevitability and coordination signal, but these are impossible to produce at the interview stage. The checks should either (a) accept interview as a valid first response, or (b) provide enough facts in the prompt to skip the interview.

4. **No em-dash violations across all 17 evals.** This was a known leak site from prior runs. The model appears to have internalized this rule.

5. **No banned-word violations across all 17 evals.** All agents correctly avoided the hard-banned list, even when quoting user copy that contained banned words.
